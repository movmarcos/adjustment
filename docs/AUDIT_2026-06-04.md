# Full-solution audit — Adjustment Engine (2026-06-04)

Cross-cutting theme: the engine repeatedly **converts failures into silent success or wrong numbers** (the same pattern as the Stress `FACT_TABLE_PK` bug). The dangerous findings are the ones that don't crash.

Severity legend: ✓ = verified against source; ⚠ = plausible, needs runtime validation.

## Confirmed correctness/data/safety bugs
1. **[Critical ✓] Netting `GROUP BY {key}` SUMs across adjustments** — `05_sp_process_adjustment.sql:837-846`. DENSE_RANK "keep latest" is defeated (one row per key already). Two adjustments on the same position add instead of supersede. Fix: `GROUP BY {key}, ADJUSTMENT_ID` so each adjustment stays a row and DENSE_RANK picks the latest; roll legs ②+③ still net (same ADJUSTMENT_ID).
2. **[Critical ✓] EntityRoll compares numeric `ENTITY_KEY` to entity-code string** (Stress/Sensitivity) — `05…:1076-1145`. Deletes target then copies zero, reports Processed → data loss. Non-atomic delete-then-copy. Fix: resolve code→key via DIMENSION.ENTITY; wrap in a transaction.
3. **[High ✓] Swallowed DIMENSION.ADJUSTMENT insert → `dim_ids_str="0"`** — `05…:554-558, 882-897`. Supersede DELETE then wipes real adjustments; batch still Processed. Fix: fail loud.
4. **[High ✓] UI treats status ≠ "Error" as success** — `1_New_Adjustment.py:1092,1112` vs `03…:31,236,434`. Signed-off "Rejected - SignedOff" shown as success. Fix: explicit success allowlist.
5. **[High ✓] Direct line items written before the SP call** — `1_New_Adjustment.py:222-235`. Orphaned on rejection/failure. Fix: clean up on non-success.
6. **[High ✓] SP_PREVIEW_ADJUSTMENT interpolates user filters unescaped** — `04…:147-182` (EXECUTE AS CALLER). Injection + breaks on apostrophe. Fix: `_esc` + int() coercion.
7. **[High ✓] Streamlit escapes SQL literal with `\'` (wrong dialect)** — `1_New_Adjustment.py:234`. Needs `''`. Fix: parameterize / `''`.

## Concurrency/design (⚠ need validation; redesign proposed separately)
- Best-effort overlap blocking proceeds on failure (`05b…:139-157`) → overlapping adjustments run concurrently, silently.
- Stream drain `WHERE 1=0` (`05b…:299-311`) can lose adjustments submitted mid-run → stuck Pending forever.
- Unguarded `collect_nowait()` submission loop (`05b…:239-246`) → one bad combo strands others in Running.
- Recommended: replace stream-as-queue + drain with atomic claim-first `UPDATE … RETURNING`.

## Same class as the fixed bug — config trusted without validation
- `validate_schema()` does not check `METRIC_NAME`, `ADJUSTMENTS_SUMMARY_TABLE`, `ADJUSTMENT_BASE_TABLE`, `DIRECT_SCOPE_SCHEMA`.
- Roll leg defaults missing combined-view **key** columns to -1/NULL; deploy passes it as a warning. For key columns this should be a hard fail.
- Misspelled fallback key `s.get("FACT_AJUSTED_TABLE")` (`05…:316`) tolerates a typo'd settings column.

## Deploy
- `deploy.py:59-61` `$$` line-parity splitter: a literal `$$` in any future proc body silently merges/drops statements (latent today).
- `CREATE OR REPLACE STREAM` on routine redeploy resets offsets → Pending work stuck.
- `--rebuild` wipes all data with no confirmation / env guard.

## UX (non-technical users)

- `current_user_name()` degrades to "unknown" and is used for the self-approval guard / audit identity.
